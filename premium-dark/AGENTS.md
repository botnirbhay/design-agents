# Premium Dark

## Style Family

**Product Storytelling and Launch.** Pair this style with Bento Motion, Editorial SaaS, Spatial 3D, and Terminal Native, but use Premium Dark when a technical SaaS launch needs dark-first credibility, code/workflow proof, infrastructure metrics, and trust controls.

Use Terminal Native instead when the whole surface is an expert command tool. Use Command Center for live operations and incidents. Use Bento Motion for broader modular SaaS proof. Use Editorial SaaS when writing, art direction, or type-led launch energy matter more than technical surfaces.

## Client-Grade Demo Target

A generated Premium Dark design must sell through credible technical artifacts. Include realistic code/config, deploy checks, regions, owners, policy table, blocked/warning states, rollback or approval path, infrastructure/security metrics, and compliance proof. The dark treatment should feel precise and expensive, not neon decoration.

## Signature Artifact Rule

Add a signature deploy evidence packet in generated examples: policy checks, regional canary state, rollback checkpoint, owner, and compliance context must be visible.

## Node Library Rules

Use Chart.js for infrastructure/security proof such as policy checks, deploy health, drift, or regional coverage. Use GSAP for precise code-panel and deploy-surface reveals, not decorative background motion.

Library use must be visible in the generated UI: include a Chart.js deploy/security proof chart and GSAP code-panel reveals that make technical evidence feel premium.

## Purpose

Premium Dark is a dark-first technical SaaS style for infrastructure, security, cloud, and developer products that need drama, precision, and real product proof.

## When To Use This Style

- Security and infrastructure launch pages.
- Developer tools with code or deploy workflows.
- Cloud products needing trust and technical credibility.
- Dark-first SaaS marketing pages with embedded product surfaces.

## Visual Identity

Near-black, graphite, and controlled accent light with sharp product artifacts. The style feels premium and serious, not neon or gloomy.

## Reference-Level Outcome

A finished page should look like a high-value technical launch where code previews, deployment health, policy checks, metrics, and trust evidence make the claim believable.

## Information Architecture

- Open with a strong dark hero and adjacent code/workflow preview.
- Follow with infrastructure or security metrics.
- Show a deployment/status surface and trust/compliance section.
- Include integrations or technical proof before conversion.
- End with a precise CTA.

## Signature Patterns

- Dark technical hero
- Code/workflow preview
- Deployment status surface
- Infra/security metric cards
- Policy or control table
- Restrained luminous CTA
- Technical trust section
- Customer or compliance proof

## Core Design Principles

- Use product artifacts before abstract illustration.
- Create hierarchy through depth, spacing, and material change.
- Use glow as a precision highlight only.
- Keep body text bright enough for long reading.
- Make code and logs realistic.
- Use semantic state colors carefully.
- Balance drama with utility.
- Never let dark surfaces collapse into one gray field.
- Pair claims with data or workflow evidence.
- Keep CTAs high contrast without neon overload.

## Layout System

- Use broad dark sections with constrained content.
- Place code or workflow previews near primary claims.
- Layer cards and panels with subtle borders and inner highlights.
- Use a rhythm of hero, proof, workflow, trust, conversion.
- On mobile, simplify panels while preserving code readability.

## Color & Theme System

- Use near-black and graphite foundations.
- Limit accent palette to one or two bright hues.
- Use soft glows and gradient strokes on priority surfaces.
- Use success, warning, and error colors consistently.
- Avoid saturating every border and label.

## Typography System

- Use sharp confident headings.
- Keep body and captions brighter than typical dark gray.
- Use monospace for code, logs, branches, IDs, and config.
- Use tabular numerals for metrics.
- Avoid thin gray labels.

## Motion Language

- Use subtle glow shifts, panel lift, code highlights, and chart draws.
- Keep motion deliberate and slower than playful UI.
- Avoid constant pulsing backgrounds.
- Use static or fade-only states under reduced motion.
- Never animate code so much it becomes unreadable.

## Component System

- Code panels need tabs, realistic syntax, and line structure.
- Metric cards need unit, trend, timeframe, and data freshness.
- Deployment surfaces show checks, regions, rollback, and owners.
- Policy tables show control, coverage, and status.
- Buttons need primary, secondary, destructive, loading, disabled, focus states.
- Badges should be semantic, compact, and readable.

## Product Realism Rules

- Use nouns like policy, deploy, trace, canary, key, region, rollback, control, audit.
- Show owners, branch names, regions, timestamps, and environment names.
- Include warning and blocked states.
- Represent secure defaults and approval checks.
- Use realistic code/config snippets and logs.

## Content Strategy

- Lead with precise technical outcomes.
- Avoid broad productivity claims without proof.
- Use concise captions explaining artifacts.
- Keep customer proof close to technical evidence.
- Use CTAs such as Start secure deploy or Book technical demo.

## Multi-Surface Requirements

- Show marketing hero, code preview, deployment/status panel, metrics, policy/trust table, and conversion surface.
- Surfaces should describe one secure workflow from policy to deploy.

## Accessibility Rules

- Audit contrast carefully.
- Use focus rings that stand out on black surfaces.
- Do not communicate status only through glow.
- Include labels on charts and code tabs.
- Respect reduced motion for animated borders and product reveals.
- Keep code readable without horizontal traps.

## Performance Rules

- Avoid heavy animated shadows and full-screen animated gradients.
- Use transform/opacity for panel motion.
- Limit glow effects.
- Lazy-load large screenshots or video if used.
- Use static alternatives for expensive backgrounds.
- Simplify effects on mobile.

## Responsive Rules

- Wrap or simplify code panels.
- Reduce glow and parallax.
- Keep primary CTAs and proof visible.
- Avoid tiny dark captions.
- Stack workflow panels in logical order.

## Best Use Cases

- Developer tools
- Security SaaS
- Cloud infrastructure
- Deployment platforms
- Technical compliance products

## Do Not Do

- Do not ship flat dark gray pages.
- Do not overuse neon.
- Do not use unreadable low-contrast copy.
- Do not fake value with abstract glowing boxes.
- Do not animate backgrounds more than product.
- Do not make warning and success ambiguous.
- Do not hide technical detail.
- Do not make all surfaces the same shade.
- Do not use vague code snippets.
- Do not omit risk states.
- Do not make CTAs glow excessively.
- Do not sacrifice readability for drama.

## Implementation Notes

- Define dark tokens for surfaces, borders, text, accents, shadows, and statuses.
- Build product artifacts as real UI compositions.
- Use semantic sample data for deploys, policies, regions, and owners.
- Keep demos static unless a real interaction needs JS.
- Use packages only for complex code editors or charts.
- Verify contrast, focus, reduced motion, and mobile code wrapping.
